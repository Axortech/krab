# Workspace Structure Standard (Krab)

## 1) Problem Statement

In Node.js ecosystems, developers usually see a clear split:

- App-owned code in `src/`
- Framework and third-party packages in `node_modules/`

For a Rust workspace, the equivalent split exists but is represented differently. This document defines a standard, discoverable structure so contributors can immediately understand what is framework-internal vs user-editable.

## 2) Rust Equivalent to `node_modules`

Krab should standardize on this mental model:

- **User-editable source**: crate-local `src/`, `tests/`, `migrations/`, `public/`, `contracts/`
- **Framework crates (editable in monorepo only)**: workspace members under a dedicated folder
- **Third-party dependencies (non-editable)**: Cargo registry cache (outside repo)
- **Build artifacts (non-editable)**: `target/` (inside repo root, gitignored)

So the practical replacement for `node_modules` is:

1. Cargo registry cache (downloaded crates)
2. `target/` dependency build outputs

Both are generated, not manually edited.

## 3) Proposed Monorepo Standard

### 3.1 Top-level layout

```text
.
├─ Cargo.toml                 # workspace manifest
├─ crates/                    # framework/tooling crates (owned by framework team)
│  ├─ framework/
│  │  ├─ krab_core/
│  │  ├─ krab_client/
│  │  ├─ krab_server/
│  │  └─ krab_macros/
│  └─ tooling/
│     ├─ krab_cli/
│     └─ krab_orchestrator/
├─ services/                  # runnable reference services/apps
│  ├─ service_auth/
│  ├─ service_users/
│  └─ service_frontend/
├─ docs/
├─ plans/
├─ scripts/
├─ .github/
└─ target/                    # generated by Cargo (gitignored)
```

### 3.2 Ownership rules

- `crates/framework/*`: framework internals and shared runtime
- `crates/tooling/*`: CLIs, generators, orchestration tools
- `services/*`: application/service implementations and adapters

This gives a stable, standard boundary similar to “framework code vs app code”.

## 4) Current-to-Target Mapping

| Current path | Target path | Ownership |
|---|---|---|
| `krab_core` | `crates/framework/krab_core` | Framework |
| `krab_client` | `crates/framework/krab_client` | Framework |
| `krab_server` | `crates/framework/krab_server` | Framework |
| `krab_macros` | `crates/framework/krab_macros` | Framework |
| `krab_cli` | `crates/tooling/krab_cli` | Tooling |
| `krab_orchestrator` | `crates/tooling/krab_orchestrator` | Tooling |
| `service_auth` | `services/service_auth` | App/Service |
| `service_users` | `services/service_users` | App/Service |
| `service_frontend` | `services/service_frontend` | App/Service |

## 5) In-Crate Standard (for every crate)

Each crate should follow a predictable layout:

```text
<crate>/
├─ Cargo.toml
├─ src/
│  ├─ lib.rs or main.rs
│  ├─ api/           # optional
│  ├─ domain/        # optional
│  ├─ adapters/      # optional
│  ├─ db/            # optional
│  └─ routes/        # optional
├─ tests/            # integration tests
├─ benches/          # optional
├─ public/           # static assets (frontend services)
└─ contracts/        # API/GraphQL/OpenAPI contracts
```

Rule: only crate-local source folders are user-editable for app behavior; generated output stays out of source trees.

## 6) Migration Plan (Low Risk)

### Phase A: Prepare (no moves yet)

1. Add this standard doc and align contributor guidance.
2. Add path policy in `CONTRIBUTING.md`.
3. Keep existing crate names unchanged.

### Phase B: Folder migration

1. Create `crates/framework`, `crates/tooling`, `services`.
2. Move folders without renaming crate package names.
3. Update workspace `members` in root `Cargo.toml`.
4. Run full CI gates (`fmt`, `clippy`, `test`, docs, deny).

### Phase C: Tooling hardening

1. Update scripts and docs for new paths.
2. Add a CI check to prevent new top-level crates outside allowed roots.
3. Add generator defaults so new crates are created in standard folders.

## 7) Standards Checklist

- [ ] No new runtime crate at repo root
- [ ] All framework crates under `crates/framework/`
- [ ] All tooling crates under `crates/tooling/`
- [ ] All runnable services under `services/`
- [ ] Root `Cargo.toml` only contains workspace-level metadata/dependencies
- [ ] `target/` and generated files remain gitignored

## 8) Why this is “standard”

- Mirrors common monorepo practice (`packages/`, `apps/`) in a Rust-native form
- Preserves Cargo conventions (`src/` per crate)
- Makes ownership and edit boundaries obvious to new contributors
- Avoids mixing framework internals with app/service code at repo root

## 9) Decision

Adopt this layout as the Krab workspace standard for all net-new crates and schedule gradual migration of existing top-level crates.
